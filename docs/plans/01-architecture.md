# Kyori Grocery Checkout Optimizer â€” Architecture Plan

**Goal:** Build an interactive checkout payment method selector that helps Kyori Grocery customers across Indonesia, Thailand, and the Philippines choose the best payment option for their situation.

**Architecture:** Single-page React application with no backend. All payment method data is mocked. State lives in `App.tsx` and flows down via props. Filtering and recommendation logic live in custom hooks. Layout is a responsive sidebar + main content + slide-out drawer.

- **No `tailwind.config.js`** â€” config lives in CSS via `@theme` directive

- **IMPORTANT:** The `@theme inline {}` block generated by shadcn is for semantic CSS variable mapping only. Do NOT mix custom tokens into `@theme inline` -- it will NOT generate utility classes. Always use a separate `@theme {}` block for brand tokens.

## 6. Filtering Logic (`usePaymentFilter`)

### Filtering Philosophy: Show "Why," Don't Just Hide

Instead of hiding methods that fail SPEED / CONVENIENCE / CATEGORY filters, incompatible methods are shown at the **end** of the grid in a "dimmed" state so the user understands **why** a method doesn't fit their situation.

- **Dimmed treatment:** `opacity-50`, not clickable for comparison
- Each incompatible card shows a badge: `"No compatible con tu entrega"` or the specific reason (e.g., "Takes up to 24 hours to confirm")
- A visual divider separates compatible from incompatible: `"Otros metodos disponibles"`

**Hard filters (hide completely):**
- **Market filter** -- methods from another country are never shown (makes no sense to display them)
- **Search filter** -- non-matching methods are removed entirely (text search is intentional narrowing)

```
filterPaymentMethods(allMethods, filterState, market):

  1. Filter by market (HARD filter â€” hides completely)
     â†’ method.markets.includes(market)

  2. Filter by search (HARD filter â€” hides completely)
     â†’ if searchQuery is non-empty, remove methods where name/tagline/category don't match

  3. For remaining methods, compute fitStatus and fitReason (SOFT filters â€” dimmed, not hidden):
     â†’ Start all as 'compatible'
     â†’ Speed check: if speed !== 'all' and confirmationMinutes > maxMinutes
         fitStatus = 'incompatible', fitReason = "Takes [time] to confirm"
     â†’ Convenience check: if convenience filter fails
         fitStatus = 'incompatible', fitReason = "Requires bank account" / "Requires store visit" / etc.
     â†’ Category check: if categories filter active and method not in categories
         fitStatus = 'incompatible', fitReason = "Not in selected categories"

  4. Sort: compatible methods first (by popularityRank ASC), then incompatible (dimmed, at bottom)

Result: single array of FilteredMethod[] â€” compatible first, then incompatible dimmed at bottom with reason visible.
```

| `'instant'` | 0 |

## 7. Recommendation Algorithm (`useRecommendations`)

Each method starts at **base score 50**. Add/subtract based on:

### Rule 1: Delivery Type Fit (max +30 / min -40)
| Delivery | Condition | Points |
|---|---|---|
| same-day | confirmationMinutes === 0 | +30 |
| same-day | confirmationMinutes <= 120 | +10 |
| same-day | confirmationMinutes > 120 | -40 |
| express | confirmationMinutes === 0 | +25 |
| express | confirmationMinutes <= 120 | +15 |
| express | confirmationMinutes > 120 | -20 |
| standard | confirmationMinutes <= 1440 | +10 |
| standard | confirmationMinutes > 1440 | -10 |
| scheduled | any | +5 |

### Rule 2: Success Rate Bonus (max +15)
```
bonus = (successRate - 0.80) Ã— 75
// 0.95 â†’ +11.25   |   0.80 â†’ 0   |   0.75 â†’ -3.75
```

### Rule 3: Popularity Bonus (max +10)
```
bonus = max(0, (6 - popularityRank) Ã— 2)
// rank 1 â†’ +10   |   rank 3 â†’ +6   |   rank 5+ â†’ +2
```

### Rule 4: Order Value Compatibility (+5 / -20)
```
if orderValue > maxAmount â†’ -20 (can't handle order size)
if orderValue < minAmount â†’ -20 (below minimum)
else â†’ +5
```

### Final Score
```
finalScore = clamp(base + rule1 + rule2 + rule3 + rule4, 0, 100)
```

Top 3 by score descending â†’ `isConfident = topScore > 60`.
